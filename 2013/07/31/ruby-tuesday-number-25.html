Ruby Tuesday #25: 意外的收獲 - Play

Jul 31, 2013

因為慕凡(@ryudo)的邀請，有機會擔任這次的 Ruby Tuesday 的講者，是一個很有趣的經驗。這次的主題是我之前寫過的文章，關於快速的部署 Rails application 在 Windows 的 IIS 上面，完整 slide 在這裡。

大概是因為主題的關係，所以真的很 Microsoft style，除了一直告訴在場的強者們如何找到一些軟體安裝一直按下一步以外好像也沒有更有趣的內容（嘿真的不要輕易嘗試把 Rails application 裝在 Windows 環境下了，* Rails HATES Windows, EXACTLY! *）

但是有趣的是在會後與Kenneth Lee的討論中，知道了 JAVA 的類 Rails MVC framework - Play，做了簡單的 study 以後有了以下的發現：

  • 也是一個 full-stack 的 Web framework
  • 它很像 Rails 學人精
  • 也可以選擇自己喜歡的 editor 而非綁住某個 IDE
  • Kenneth 同時提到另外一個優點，Heroku 也支援 Play

感覺減輕了相當多開發 JAVA 的痛苦，並且它應該是在模仿動態語言的特性，也模仿了 Rails 的哲學，可以自動載進 JVM 不需要像以往一樣 restart，就可以看到新的修改，整個的架構也是 follow Rails，同樣有慣例優於配置的原則，這對 Rails developer 來說真的是接觸 JAVA Web application 更無痛的選擇。

我覺得自己是被 Ruby 輕鬆自然的語法糖給慣壞了，但是實際的專案中，一定會碰到大大小小非 Ruby 語言或是 Rails 框架的老專案，這些專案的生命週期很長，最可怕的是當年製造出 code 的人總以為這是一個可能一年兩年就不要了的小案子，寫出來的東西就是能動就好了，基本上要碰到會長遠去規劃程式架構的 source code，完全就是看你上輩子燒的香有沒有夠。我喜歡 Ruby，喜歡寫 Rails，然而對於每分鐘都收進大量數據刷資料庫的交通類應用我怎麼考慮都感到不甚適合，因為一來是 Rails 並非一個很快速的架構，再來就是也不能發揮 Rails 的優點。

針對一件事情去考慮好用的工具也是實作時很重要的部分，例如@tka在當晚的講題以 JRuby + Mahout 去實作推薦系統就是一例，怎麼樣去擇定適合這件事的語言與框架，將部署環境等等也考慮在內，現實來講就是工程師必須去貼合那些需求（為了成本和利潤甚至合約要求等等因素），而不能隨心所欲的總是使用偏好的或自己更熟稔的語言，這也是我對於突破舒適圈這件事感到如此重要的原因，要去解決那些困難，要學會去選擇工具，就必須時時的突破舒適圈，學習更多未知的知識與技術。
